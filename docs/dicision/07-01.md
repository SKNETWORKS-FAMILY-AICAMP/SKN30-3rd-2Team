## 1. 매칭 점수 스케일 결함 규명 — 전건 오작동의 근본 원인

- **진단: `hybrid_search`가 반환하는 `fusion_score`(RRF)는 이론상 최댓값 ≈0.033(=2/61)인데, 이를 `match_threshold=0.5`와 직접 비교했다. 또한 아키텍처가 전제한 리랭커 단계가 코드에서 통째로 누락됐다.**
- **결과:** `select_best_match`가 항상 `(None, 0.0)` 반환 → 모든 사용자 조항 `NO_MATCH`, matched_ids 공집합 → 모든 표준조항 `MISSING`. 테스트가 통과한 건 `FakeRetriever`가 `rerank_score=0.95`를 하드코딩했기 때문.
- **불변식 확정:** 매칭 판정에는 **반드시 Reranker를 통과한 `rerank_score`만** 쓴다. `fusion_score`(랭크 기반)는 threshold와 스케일이 달라 매칭 비교에 절대 쓰지 않는다.

## 2. 매칭 점수 정규화 — 리랭커 + sigmoid *(둘 다 채택)*

- **결정: `rerank_score`(BGE 리랭커의 raw 로짓)를 `core.sigmoid`로 0~1 정규화해서 threshold와 비교한다.**
- **이유:** `CrossEncoder.predict`는 정규화되지 않은 로짓(음수~양수)을 반환한다. sigmoid(로짓 0)=0.5 이므로 **`match_threshold=0.5` = "모델이 양(+)의 판단"** 이라는 해석이 유지된다.
- **배치:** `sigmoid`는 순수함수라 `core/matching.py`에 두고 export(오버플로 방지 위해 부호 분기). 정규화 지점은 `pipe`(조립).
- **기각 안 함:** "fusion_score 정규화" 단독안(리랭커 미사용)은 매칭 정밀도가 낮아, **리랭커 도입 + sigmoid 정규화를 함께** 채택.

## 7. 배치 검색 — `Retriever.search_many` 신설

- **결정: 조항별 개별 `search`(N회)를 컬렉션당 1회 `search_many`로 축소한다. 포트에 `search_many`를 노출하고 `VectorManager`에 concrete 구현.**

- **구조:** `_dense_query`를 분리해 미리 계산한 벡터를 재사용. "port를 통해 배치를 노출" → 헥사고날 유지. 파이프는 컬렉션당 1회 호출, rerank는 조항별 유지(§10).

## 9. sub-chunk 메타데이터 확장 — contract_type만 채택, category 보류

- **결정 (contract_type 추가):** 서브청크 Chroma 메타데이터에 `contract_type`을 추가한다.
  - **이유:** contract_type은 **파티션 키**다 — 문서 전체가 결정하며 자식 항이 다른 유형이 될 수 없어 **무드리프트**. 혼합 컬렉션(`standard_sub_chunks`는 전 유형 적재)에서 유형 필터 없이 `top_k`를 뽑으면 타 유형이 진짜 부모를 밀어내 **recall이 깨진다**(효율이 아닌 정확도 문제). 검색 단계 `where` 필터로 사후 부모매핑 보정 의존도 제거.
  - 파이프는 서브청크 `search_many`에 `type_filter`를 넘김.
- **결정 (category 보류):** 서브청크에 category/source는 **추가하지 않는다**(현행 roll-up-to-parent 유지).
  - **이유:** 롤업 후 부모 `StandardClause`가 이미 category를 보유 → 서브청크 category는 **소비처 없는 잉여**. contract_type과 달리 "세부 항이 다른 성격"이 실재해 부모값 복사(materialize)는 **드리프트 위험**. 이는 "서브청크 단위 라우팅으로 갈 것인가"라는 조율된 결정과 묶이므로 별도 사인오프 대상.
  - **조건부:** 정말 서브청크 라우팅으로 간다면 (a) `roll_up_sub_chunks`가 category 보존/방출 (b) 메타데이터 추가 (c) 03_normalized 수작업 라벨링이 필요하며, 넣더라도 `category: Optional[Category] = None`(override 시에만 저장, 복사 금지).

## 10. 리랭킹 배치화 — `rerank_many` 시임만 구현 (파이프 미적용)

- **결정: 포트에 `Reranker.rerank_many`를 추가하고 구현체까지 두되, 파이프는 당분간 조항별 `rerank` 유지. 나중에 드롭인만 하면 되도록 준비.**
- **구현:** 로컬 `BgeReranker`는 전 질의의 (질의,문서) 쌍을 **flatten → `model.predict` 1회 → 순수 헬퍼 `_scatter_reranked`로 질의별 되담기**. API `ApiReranker`는 엔드포인트가 per-query 규격이라 rerank 루프(동작 동일, 네트워크 배치 불가).
- **이유:** cross-encoder는 쌍 단위 채점이라 질의별 호출이 자연스럽지만, flatten 배치는 모델 배치를 채워 호출 오버헤드를 줄인다. **각 쌍은 독립 채점 → 점수 불변(성능만 개선)**. 임베딩 왕복(주 병목)은 이미 §7로 잡혔고 rerank는 조항당 top_k(~5)만 보므로, **프로파일링에서 rerank가 병목으로 뜰 때** 파이프 교체.

## 13. 2트랙 평가 구조 — 실제 vs 합성 *(채택)*

- **결정: 골든셋을 두 트랙으로 나눈다. 두 트랙 모두 같은 `metrics.py`로 수렴하고, driver만 트랙별로 cases를 만든다.**
  - **트랙 A(합성):** 케이스 1건 = 사용자 조항 1개 = **독립 검색 질의**. Recall@k·MRR·**ablation 4변형**·이탈 분류 P/R·독소 P/R. 함정(paraphrase/contradiction/partial) 집중. → 현행 `*.example.json` 92건.
  - **트랙 B(실계약):** 단위 = 계약서 문서(`clauses[]`+`expected_missing[]`). `parse_contract`→**`review_contract` 전체 파이프**→사람 라벨 비교. 이탈 P/R·**MISSING Recall**·현실 성능.
- **이유:** `MISSING`(표준에 있는데 계약 전체에 없음)은 **계약 단위**라 조항 단위 합성셋으로 측정 불가(합성 92건에 MISSING 0건은 버그가 아니라 구조). 반대로 ablation(RAG 정당성 대표 결과물)은 함정 밀도 높은 합성 조항 질의가 적합. → 두 관심사를 트랙으로 분리.

## 14. 이번 세션 범위 — 트랙 B 보류, 문서+설계까지 *(사용자 결정)*

- **결정: 트랙 B는 보류.** 실계약(또는 문서 단위 합성 계약서) + 사람 라벨이 아직 없어 MISSING/E2E는 **스키마만 남기고** 미룬다.
- **결정: 이번 세션은 문서+설계까지.** D_eval.md 재작성·결정 로그 확정. driver 재작성 및 이탈/독소 P/R 하니스 **코드 구현은 다음 단계**.

## 15. 골든셋 정합성 결함 수정 — 독소 라벨 enum 정합 *(사용자 결정: 골든을 enum에 맞춤)*

- **발견(실측):** 합성 골든의 `gold_toxic`에 **`ToxicPattern` enum에 없는 라벨 2종** — `UNFAIR_TERMINATION`(sw g09), `UNPAID_COMPENSATION`(sw g15). enum·`toxic_patterns.json` 코퍼스 어디에도 없어 **탐지 원천 불가**였다.
- **결정: 동결 계약(enum)은 건드리지 않고 골든 라벨을 기존 enum 값으로 재매핑.** 둘 다 → **`UNILATERAL_CANCELLATION`**(일방적 계약 취소·무보상 해지).
  - **이유:** g09(기대미달 부당해지+전액환수)·g15(해지 시 기수행 보수 포기)는 `toxic_patterns.json`의 `UNILATERAL_CANCELLATION` 코퍼스("무보상 계약 해지")와 의미 일치. enum 확장(동결 변경·사인오프)보다 라벨 정정이 저비용·무위험.
  - **검증:** 전 골든 `gold_toxic` enum 정합성 스캔 → OUT-OF-ENUM 0. 사용 패턴 6종 전부 enum·코퍼스 존재.

## 16. trap 스키마 정식화 — `contradiction` 추가·`reorder` 유지 *(채택)*

- **결정: trap 유형을 `none`/`paraphrase`/`reorder`/`partial`/`contradiction` 5종으로 정식화.**
  - `contradiction`(조항번호 언급하나 내용 반대 — 랭킹 교란)은 실측 19건으로 주력인데 스키마 미문서 상태였다 → 정식 편입.
  - `reorder`(항·호 순서 뒤바뀜)는 기획서 8.3 함정 3대 축이나 **현재 0건** → 스키마엔 유지, **골든 확장 시 보강**(백로그).

## 17. Driver 재작성 규격 확정 — rerank 포함 4변형 *(설계)*

- **결정: 낡은 `eval/run_eval.build_cases`를 재작성한다.** 결함: `from src.adapter import vector`(컨벤션 위반), `standard_clauses`만 검색, **rerank 미호출**(07-01 §1 불변식 "매칭은 rerank_score만"과 배치되고 `hybrid_rerank` 변형 불가).
- **변형 생성:** `bm25`/`dense`/`hybrid` = `vector.search(search_type=…)`; `hybrid_rerank` = hybrid pool(≈k×4) → `reranker.rerank(q, pool, top_k=k)`. EXTRA(`gold_clause_id=null`)는 검색 정답 없음 → Recall@k/MRR 집계 제외.
- **이탈/독소 P/R(A-2/A-3):** 케이스 조항 1개를 `review_contract([clause], ct)`에 태우고 `user_clause` 일치 결과만 취함(단일 조항이라 나머지 표준은 MISSING 노이즈 → 버림). 이진 이탈(deviation≠NONE) vs `gold_deviation≠NONE`, 독소는 `toxic_patterns` 비어있음 여부 vs `gold_toxic` 존재 → `precision_recall`.
