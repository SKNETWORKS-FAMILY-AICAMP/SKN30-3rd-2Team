## 0. 세션 결론 (한 줄)

RunPod Hub 워커는 rerank가 근본적으로 깨져 있어 기각하고, **`src/adapter/embedding_model.py`를 그대로 재사용하는
자체 서버리스 워커**(`deploy/runpod_worker/`)를 신설·배포했다. 모델은 `--model-reference`/네트워크 볼륨이 아니라
**빌드타임에 이미지에 직접 굽는 방식**으로 캐싱하고, ghcr.io에 public 이미지로 push했다. 운영은 `workers-min`
토글(작업 전 1 → 후 0)로 콜드스타트 없이 비용 0원을 유지한다. 추가로 `ApiReranker.rerank_many`의 질의당
N회 네트워크 호출을 **배치 1회 호출**로 리팩터링했다(Protocol 시그니처는 무변경).

## 1. RunPod Hub `worker-infinity-embedding` 기각 — rerank 미해결 버그

- **발견: 이 워커의 `handler.py`는 rerank 결과(`RerankReturnType` pydantic 객체)를 `model_dump()` 없이 그대로
  반환해 RunPod가 JSON 직렬화하다 실패한다.** 모델·설정과 무관하게 항상 재현됨(README 예제 curl도 실패).
  GitHub에 동일 증상 이슈 2건이 장기간 open 상태([#37](https://github.com/runpod-workers/worker-infinity-embedding/issues/37) 2026-01-16,
  [#29](https://github.com/runpod-workers/worker-infinity-embedding/issues/29) 2025-05-23), 메인테이너 응답 없음, 병합된 PR 없음.
- **이유로 기각:** 임베딩 경로는 정상 동작하나 rerank가 이 프로젝트 매칭 로직의 핵심(07-01 §1 불변식: 매칭은
  `rerank_score`만 사용)이라 대체 불가한 결함.

## 2. Pod 전환 기각 — 근본 원인이 배포 방식과 무관

- **배경: 아시아 리전에 저사양 GPU가 없어 EU/NA GPU를 쓰는데 느리다는 문제 제기 → Pod로 전환해 "쓸 때만
  켜고 안 쓸 때 삭제"하는 방식을 검토.**
- **기각 사유 1 (기술적):** `worker-infinity-embedding` 이미지는 `runpod.serverless.start()`로 잡 큐를 폴링하는
  서버리스 전용 워커라 `EXPOSE` 포트가 없다 — Pod로 띄워도 호출할 방법이 없다. Pod로 가려면 아예 다른 이미지
  (Infinity 엔진 자체, `michaelf34/infinity`)가 필요.
- **기각 사유 2 (근본 원인 무관):** `runpodctl datacenter list`로 확인한 결과 아시아 데이터센터(`AP-IN-1/2`,
  `AP-JP-1`, `SEA-SG-1`)엔 H100/H200급만 있거나 GPU 재고가 아예 없다 — **Pod든 서버리스든 동일한 RunPod
  재고 문제**라 배포 방식을 바꿔도 EU/NA를 써야 하고 네트워크 왕복 지연은 그대로 남는다.
- **최종 채택: 서버리스 유지 + `workers-min` 토글**(작업 세션 전후로 0↔1). 콜드스타트(진짜 원인으로 추정)만
  제거하면 되고, Pod 전환의 실익이 없다.

## 3. vLLM 검토 후 기각

- **검토 계기:** 커스텀 워커 대안으로 vLLM 제안됨.
- **기각 사유:** RunPod 공식 `worker-vllm` 허브 이미지는 텍스트 생성(chat/completion) 전용이라 임베딩/리랭킹
  엔드포인트가 없음. vLLM 엔진 자체는 pooling/score 태스크를 지원하지만 **모델 1개당 서버 1개**만 서빙해
  임베딩·리랭커를 각각 별도 인스턴스로 띄워야 하고, RunPod에 올리려면 handler.py를 처음부터 새로 짜야 해서
  1줄 패치보다 작업량이 크다. 한국어 파인튜닝 체크포인트(`dragonkue/*`)의 vLLM pooling/classify 호환성도
  커뮤니티 검증 사례가 없다.

## 4. 커스텀 워커 채택 — 기존 어댑터 코드 재사용 *(핵심 결정)*

- **결정: `worker-infinity-embedding` 포크·패치 대신, `src/adapter/embedding_model.py`(`Bgem3Embedder`/
  `BgeReranker` 싱글톤)를 그대로 재사용하는 자체 RunPod 서버리스 워커를 새로 만든다.**
- **이유:** (a) 포크 패치안은 서드파티(별 47개, open issue 9개, 사실상 방치)에 계속 의존한다. (b) 자체 워커는
  로컬/운영 코드가 100% 동일해 동작 불일치 위험이 없다. (c) RunPod Hub 전체에서 임베딩/리랭킹 카테고리는
  이 워커 하나뿐(TEI 등 대안 리스팅 없음, `runpodctl hub search` 직접 확인).
- **구현:** `deploy/runpod_worker/`(`handler.py`·`Dockerfile`·`test_input.json`·`README.md`) 신설.
  `handler.py`는 `adapter` 패키지 `__init__.py`(kordoc·korean_law_mcp·db 등 무거운 의존성 포함)를 거치지
  않도록 `src/config.py`·`src/adapter/embedding_model.py`를 빌드 시 flat 복사해 import한다.
- **job input/output 스키마:** 기존 `src/adapter/api_embedding_model.py`(`ApiEmbedder`/`ApiReranker`)의
  호출 규격과 1:1로 맞춰, 그쪽 어댑터 코드는 **무수정**으로 이 워커를 그대로 호출한다.

## 5. 모델 가중치 베이킹 — `--model-reference`/네트워크 볼륨 기각, 빌드타임 내장 채택

- **1차 시도 실패: `HF_HOME=/runpod-volume/huggingface-cache` + `--model-reference`로 배포했으나 워커가
  `EXITED`로 죽고 재시도도 안 됨.** 네트워크 볼륨을 안 붙인 엔드포인트라 `volumeInGb: 0`이었고,
  `/runpod-volume` 경로가 실제 쓰기 가능한 디스크가 아니었을 가능성이 원인으로 추정됨(로그 트레이스백까지는
  확보 못함, 정황 증거 기반 진단).
- **결정: [RunPod 공식 커스텀 템플릿 가이드](https://docs.runpod.io/pods/templates/create-custom-template)의
  권장 방식대로, 모델 가중치를 `docker build` 시점에 이미지 안에 직접 다운로드해 굽는다**
  (`Dockerfile`에 `RUN python -c "SentenceTransformer(...); CrossEncoder(...)"`, `HF_HOME=/app/models`).
- **이유:** 네트워크 볼륨·`/runpod-volume` 가용성이라는 불확실한 전제에 기대지 않고, 이미지 자체에 가중치가
  포함되어 있어 콜드스타트마다 재다운로드가 필요 없다. `--model-reference` 플래그는 이후 배포에서 제거.

## 6. 레지스트리 — ghcr.io, 토큰 스코프·attestation 매니페스트·visibility 이슈 순차 해결

- **레지스트리: ghcr.io(GitHub Container Registry) 채택**, `.env`의 `GITHUB_TOKEN` 사용.
- **문제 1 — 토큰 스코프:** 최초 토큰에 `write:packages` 없어 push 실패(`permission_denied`) → 사용자가
  스코프 포함해 토큰 재발급 후 해결.
- **문제 2 — buildx attestation 매니페스트:** `docker build` 기본값(provenance/SBOM 첨부)이 GHCR와 호환
  안 돼 블롭은 다 올라가고 최종 매니페스트에서 `unauthenticated` 발생 → **`--provenance=false --sbom=false`
  로 빌드**해 해결(이후 모든 빌드에 고정 적용).
- **문제 3 — 패키지 visibility:** ghcr.io 신규 패키지는 기본 private. RunPod가 pull 시 `IMAGE_AUTH_ERROR`
  (`unauthorized`). `runpodctl registry create`로 registry auth(`workshield-ghcr`)를 만들어봤지만 CLI에
  템플릿↔registry-auth 연결 플래그가 없어 자동 매칭 안 됨. GitHub REST API로 visibility PATCH 시도했으나
  **GitHub가 컨테이너 패키지의 visibility API 변경 자체를 막아둠**(404) → **웹 UI에서 수동으로 public 전환**해
  해결. (registry auth 엔트리는 현재 미사용으로 남아 있음 — 추후 private 이미지가 필요해지고 RunPod가
  템플릿-레지스트리 연결을 CLI로 지원하면 재사용 가능.)

## 7. 운영 워크플로 — `workers-min` 토글 + 클라이언트 타임아웃 상향

  → **180초로 상향**(`api_embedding_model.py:17`).

  `ghcr.io/hong1008/workshield-embed-rerank:latest`). `.env`의 `RUNPOD_ENDPOINT_ID` 갱신 완료.
  (※ 중간에 만들었던 구 엔드포인트 `f8h18nv8vt1qn1`/`lepnul78jmuk3d`와 템플릿 `odm5f3rwua`는 트러블슈팅

## 8. Rerank 배치 API — 질의당 N회 네트워크 호출 → 1회로 축소 *(호출 사양 변경)*

- **문제 인지: `ApiReranker.rerank_many`는 질의마다 `self.rerank()`를 호출해 RunPod에 순차적으로 N번 네트워크
  왕복을 하고 있었다.** 07-01 §10 결정 당시엔 "엔드포인트가 per-query 규격이라 배치 불가"가 이유였는데,
  이제 자체 워커를 소유하므로 그 제약이 사라졌다는 지적이 나와 재검토.
- **결정: 배치 rerank job 스키마를 추가한다.** `{"queries": [...], "docs_per_query": [[...], ...]}` →
  `{"scores_per_query": [[...], ...]}`(정렬 없음, 입력 순서 유지 — 정렬·top_k 절단은 여전히 호출측 책임).
- **구현:**
  - `BgeReranker.compute_scores_many` 신설(`src/adapter/embedding_model.py`) — 전 질의의 (질의,문서) 쌍을
    flatten 해 `model.predict` 1회로 채점하되, `rerank_many`/`_scatter_reranked`와 달리 **정렬하지 않고
    원본 순서 그대로 반환**(네트워크 계층이 원시 점수만 넘기기 위함). 기존 `rerank_many`는 이미 로컬에서
    단일 predict 호출로 효율적이라 **변경하지 않음**(리스크 없는 범위만 수정).
  - `deploy/runpod_worker/handler.py`에 `queries`(복수형) 라우트 추가 — `reranker.compute_scores_many` 직접 호출.
  - `ApiReranker.rerank_many`(`src/adapter/api_embedding_model.py`) 리팩터링: N회 순차 호출 → `compute_scores_many`
    **1회 배치 호출**로 대체. `Reranker.rerank_many`의 Protocol 시그니처(`adapter/port.py`)는 **무변경** — 동결
    계약 위반 없음.
- **검증:** 모킹 테스트로 순서 보존·질의별 그룹핑 정확성·네트워크 호출 횟수(2질의 → 1회) 확인 후, 실제
  엔드포인트에서 2질의 배치 rerank 성공 확인(정답 조항 각각 0.99+, 무관 조항 ~0).

## 후속(미결) 항목

- **`workers-min` 토글 자동화** — 현재 수동(`runpodctl serverless update`); justfile 레시피(`embed-on`/
  `embed-off`)로 감싸는 안을 제안했으나 이번 세션엔 미적용.
- **레지스트리 private 전환 재검토** — 현재 ghcr.io 이미지는 public(시크릿 미포함이라 위험 낮음). RunPod가
  템플릿-레지스트리 연결을 CLI로 지원하게 되면 만들어둔 `workshield-ghcr` registry auth로 private 전환 검토.
- **`docker run --test_input` 로컬 스모크 플로우 미검증** — README에 기술만 하고 실제로는 원격 배포 후
  RunPod 엔드포인트에서 직접 검증함. 로컬 CPU 스모크 경로는 실행해본 적 없음.
