## 1. Z_Lever 설계 전면 철회

- **결정: 구 Z_Lever(element 정렬·4대 법률 레버 룰·심각도 결정표) 설계 폐기.**
- **이유:**
  - element 정렬 → 1:N/N:1 허용을 위해 임계값 다대다 매칭을 채택해도, 표준 element 별 정렬·레버 비교까지 가면 "어디가 어떻게 불리한가" 해석이 1차 코드 안으로 들어온다 → 규칙 #1(1차 LLM·해석 금지) 경계 위반.
  - 동결 계약 변경 3건(granularity·change_kind·ElementResult 신설) + standard_elements 테이블·신규 enum 승인이 선행 조건이었으나, 구조 복잡도 대비 효용이 검증되지 않음.
  - **1차 MVP의 역할은 "어느 조항이 CHANGED되었는지 정확히 탐지"이며, "어디가 어떻게 불리한지"는 2차 LLM 역할.** Z_Lever는 이 경계를 침범했음.
- **조치:** 동결 계약 변경 승인 안건 전면 철회. [G_sub_chunk.md](../tasks/G_sub_chunk.md)로 대체.

## 2. 임베딩 희석 문제의 정확한 분석

- **결정: "임계값(`change_threshold`)을 올리면 항 삭제를 탐지할 수 있다"는 가정을 폐기.**
- **이유:** 10개 항 중 1개 삭제 시 나머지 9개가 표준과 일치 → 전체 유사도 ≈ 0.97. 임계값을 0.99로 올려도 통과. 임계값은 marginal 케이스 보조 수단이지 삭제 탐지 메커니즘이 될 수 없음.
- **채택: 서브청크 커버리지 체크**가 삭제 탐지의 주 메커니즘.

---

## 3. Roll-up 스코어 집계 방식 — Max 확정

- **결정: 부모 조항 확보(retrieval) 단계의 롤업은 Max Score로 고정. Mean은 기각.**
- **이유:** 거대 조항에서 사용자가 특정 항만 언급할 경우, Mean은 나머지 항의 낮은 점수에 묻혀 올바른 부모를 놓친다. Max("하나라도 잘 맞으면 후보 부모")가 retrieval 목적에 맞음.
- **Mean이 필요한 상황 없음:** 롤업은 부모 확보 전용. 삭제 탐지는 이후 커버리지 체크(불리언)가 담당하므로 여기서 평균을 낼 이유가 없어짐.

---

## 4. 커버리지 체크 — 삭제 탐지의 주 메커니즘 채택

- **결정: 부모 매칭 후, 해당 부모의 표준 서브청크 각각에 대해 사용자 서브청크 중 임계값 이상 매칭이 하나라도 있는지 확인. 미커버 서브청크 존재 시 → `Deviation.CHANGED` 상향.**
- **기존 로직과 방향이 반대임을 명시:**
  - 기존(retrieval): 사용자 → 표준 서브청크 탐색
  - 커버리지 체크: 표준 서브청크 각각 → 사용자 서브청크에서 대응 확인
- **장점:** Z_Lever의 element 정렬 없이도 항 삭제를 결정론적으로 탐지. 동결 계약 변경 불필요.

---

## 5. 커버리지 체크 범위 — 조 내부(clause-local)로 한정

- **결정: 커버리지 체크는 해당 조 내 사용자 서브청크만 대상으로 한다. 전체 계약 교차 탐색(option b)은 채택하지 않는다.**
- **판단 배경:** 사용자가 표준 ⑩을 다른 조에 흩어 적었을 경우, 조 내부 커버리지만으로는 "삭제"와 "재배치"를 구분할 수 없다.
- **이유 (option b 기각):**
  1. 전체 교차 탐색도 정답을 보장하지 않는다 — 다른 조의 유사 문구가 오매칭될 수 있어 오히려 노이즈 증가.
  2. "흩어짐이 허용되는가"는 조문 위치·효력 해석의 문제 → 2차 LLM이 계약 전체 맥락에서 판단해야 할 영역. 1차가 흡수하면 규칙 #1 경계를 침범.
- **결론:** "표준 ⑩ 미커버 → CHANGED → 2차로 넘김" 흐름이 1·2차 역할 분담의 정확한 구현. option a의 false positive는 2차 입장에서 valid input.

## 8. 평가 지표 명확화

- **결정: G 고도화의 주 측정 지표를 Coverage Recall로 명확화.**
- **정의: 삭제된 표준 서브청크를 "미커버"로 잡아내는 비율.**
- **이유:** 기존 Recall@k(retrieval용)는 "올바른 부모 조항을 찾는가"를 측정하며, "삭제된 항을 탐지하는가"와는 다른 지표. G의 핵심 기여는 항 삭제 탐지이므로 Coverage Recall이 주 지표.
- **LLM-judge 미도입** (규칙 #5 준수), F1·Recall 등 결정론적 수치만 산출.

## 11. 임베딩·리랭커 모델 — HF Serverless API + 리랭커 no-op

- **결정: 시연 기간 동안 임베딩은 HF Inference API, 리랭커는 no-op(통과)으로 처리.**
- **이유:**
  - bge-reranker-v2-m3 전용 rerank 엔드포인트가 HF Serverless API에 없음. text-classification으로 우회 시 배치 처리 복잡도 증가.
  - 시연 규모(수백 건 조항)에서 리랭커 제거로 인한 정확도 저하가 미미.
- **구현:** `adapter/reranker.py` no-op 구현 — candidates 그대로 반환.
- **주의:** HF API 콜드 스타트 ~20초. 시연 전 워밍업 호출 필요.
- **전환 조건:** 실서비스 또는 정확도 검증 시 infinity-emb VM으로 교체. `adapter/embedder.py` 모델명·엔드포인트 환경변수만 교체하면 됨.

## 12. bge-m3 선택 근거 재검토 — dense 단독 사용 확인

- **결정: 현재 구현이 bge-m3의 dense 출력만 사용함을 확인. bge-m3 고유 이점(sparse·ColBERT)이 활용되지 않고 있음.**
- **분석:**
  - Dense: Chroma에 저장 → 사용 중.
  - Sparse (lexical weight): BM25(Kiwi)로 대체 중 → bge-m3 sparse 미사용.
  - ColBERT (multi-vector): Chroma가 문서당 벡터 1개만 지원 → 구조적으로 저장 불가.
- **결론:** dense만 쓰면 bge-m3는 크고 느린 일반 임베딩 모델과 동일. 경량 한국어 모델과 비교 평가 필요.

## 13. 임베딩 모델 비교 평가 계획

- **방법:**
  1. `data/03_normalized/standard_clauses.*.json` 기반 테스트 쌍 20~30개 수작업 구성 (`eval/clause_pairs.json`).
  2. 리랭커 없이 dense 검색 단독으로 비교 (BM25는 모델 무관이므로 변수 격리).
  3. `eval/compare_embedders.py` 스크립트로 Hit@1·Hit@3 산출.
- **판단 기준:**

  | Hit@1 차이 | 결론 |

  | 10% 초과 | bge-m3 유지 |

- **후속:** 비교 결과에 따라 `adapter/embedding_model.py` 모델명만 교체.

## 14. Chroma 다중 인덱스 — collection_name 파라미터로 이미 지원

- **결정: `VectorManager`가 이미 `collection_name` 인자 기반으로 다중 컬렉션을 지원함을 확인. 추가 설계 변경 불필요.**
- **구조:**
  ```
  PersistentClient
  ├── "standard_clauses"   ← 표준조항
  └── "toxic_patterns"     ← 독소패턴 (build_toxic_index 추가만 하면 됨)
  ```

- **BM25 캐시도 `(collection_name, filter_tuple)` 키로 컬렉션 간 격리됨** — 기존 코드 수정 없이 컬렉션명만 다르게 넘기면 독립 인덱스로 동작.

## 15. 포트 위치 결정 — 도메인 포트 vs 어댑터 포트 분리

- **결정: `Embedder`·`Retriever`를 `contracts/ports.py`에서 `adapter/port.py`로 이동. `Reranker`·`Database` 신규 포트도 `adapter/port.py`에 추가.**
- **이유:**
  - `Embedder`·`Retriever`는 벡터 DB·모델 구현에 의존하는 인프라 계약이므로, 도메인 무관 계약(`Parser`·`Grounder`·`Graph`)이 모인 `contracts/`가 아닌 `adapter/`에 귀속시키는 것이 헥사고날 경계에 부합.
  - `contracts/ports.py`는 도메인 행위(파싱·법령 조회·그래프 탐색)만 남겨 순수 도메인 계약으로 유지.
- **기각된 옵션:** `contracts/infra_ports.py` 별도 파일 — 패키지 경계(`contracts` vs `adapter`)가 이미 역할을 표현하므로 불필요한 파일 분할.
- **`markdown_splitter`·`kordoc`·`korean_law_mcp`는 포트 없이 유지:** 각 어댑터가 고유 기능 집합을 가져 Protocol 추상이 실익이 없음. 상위 포트(`Parser`·`Grounder`)가 이미 도메인 경계를 잡고 있어 테스트 격리 충분.

## 16. VectorManager — Embedder 생성자 주입 리팩토링

- **결정: `VectorManager.__init__`에 `embedder: Embedder` 파라미터를 추가하고, 내부 싱글톤 직접 import를 제거. 조립 지점을 `adapter/__init__.py`로 일원화.**
- **이전 구조 문제:** `from .embedding_model import embedder`를 모듈 레벨에서 고정 import → `VectorManager` 단독 테스트 시 실제 임베딩 모델이 메모리에 로드됨.
- **변경 후:** `vector = VectorManager(embedder=embedder)` 조립을 `adapter/__init__.py`에서 수행. 테스트에서는 `VectorManager(embedder=MagicMock(spec=Embedder))`로 모델 로드 없이 격리 테스트 가능.
- **기존 사용 인터페이스 무변경:** `from adapter import vector` 패턴은 동일하게 유지.

## 17. 운영용 임베더·리랭커 — RunPod Serverless API 채택

- **결정: `app_env != "local"` 환경에서는 `adapter/api_embedding_model.py`의 `ApiEmbedder`·`ApiReranker`를 사용. RunPod `/runsync` 엔드포인트를 HTTP로 호출.**
- **구현 방식:**
  - `ApiEmbedder.embed_documents` → POST `/runsync`, `output.data[].embedding` 추출 후 `index` 정렬로 입력 순서 보장.
  - `ApiReranker.compute_scores` → POST `/runsync` (`return_docs: false`), `output.scores[]` 추출.
  - RunPod 응답 외피(`output` 키) 언래핑은 `_call_runsync()` 내부에서 처리 — 호출부에 노출 안 됨.
- **콜드 스타트 대응:** `requests.post` 타임아웃 60초 고정 (RunPod cold-start 최대치).
- **환경변수:** `RUNPOD_API_KEY`·`RUNPOD_ENDPOINT_ID` — `config.py`에 추가, `.env`로 관리.

## 18. app_env 기반 구현체 선택 — adapter/__init__.py 조립

- **결정: `adapter/__init__.py`에서 `app_env == "local"` 분기로 로컬 모델·API 구현체를 선택. 외부 소비자 코드는 변경 없음.**

  ```python
  if app_env == "local":
      from .embedding_model import embedder, reranker
  else:
      from .api_embedding_model import api_embedder as embedder, api_reranker as reranker
  ```

- **이유:** 환경별 구현체 교체가 단일 지점(adapter 조립)에서 일어나므로, pipe·server·core는 `app_env` 를 알 필요가 없음. 헥사고날 원칙의 "코어는 외부를 모른다" 유지.
- **전환 조건:** `.env`의 `APP_ENV` 값 변경만으로 로컬 ↔ 운영 전환 가능.
