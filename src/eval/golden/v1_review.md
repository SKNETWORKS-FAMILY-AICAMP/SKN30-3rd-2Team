## 0. 먼저 — 지표 용어 빠르게 (이 문서 이해에 필요)

각 조항을 "이탈이다/아니다"(또는 "독소다/아니다")로 맞히는 문제라, 채점은 4칸으로 갈립니다.

| | 실제 양성(정답=이탈) | 실제 음성(정답=정상) |
| --- | --- | --- |
| **예측=양성** | TP (맞게 잡음) | **FP (정상을 이탈로 오판)** |
| **예측=음성** | FN (놓침) | TN (정상을 정상으로 통과) |

- **Recall(재현율)** = TP / (TP+FN) → *진짜 이탈 중 몇 %를 잡았나* (놓치지 않는 능력)
- **Precision(정밀도)** = TP / (TP+FP) → *이탈이라 외친 것 중 몇 %가 진짜였나* (헛알람 안 내는 능력)
- **Specificity(특이도)** = TN / (TN+FP) → ***정상 조항을 정상이라고 통과시키는 능력*** ← v1의 핵심 문제
- **base rate** = 골든셋에서 원래 양성(이탈)인 비율. 전부 "이탈"이라 찍으면 precision은 자동으로 이 값이 됩니다.

> 왜 특이도가 중요? Recall만 보면 "다 이탈이라고 찍기"만 해도 **Recall=100%**가 나옵니다(안 놓치니까).
> 그게 실력인지 그냥 다 찍은 건지는 **특이도**로 구분합니다. 특이도가 0이면 정상 조항을 하나도
> 못 통과시킨 것 = **아무거나 다 이탈이라고 한 것**입니다.

## 1. 한 줄 요약

- **검색(A-1)은 정상.** dense recall@5=0.953이고, 리랭커가 순위 지표 MRR을 0.841→0.886으로 끌어올려 RAG의 가치가 수치로 확인됩니다.
- **분류(A-2/A-3)는 고장(축퇴).** 이탈·독소 모두 Recall=1.0인데 **특이도=0**입니다. 즉 "재현율 100%, 완벽!"이 아니라 **검토된 모든 조항을 이탈·독소로 찍고 있는 것**입니다. 정상 기준선 케이스(예: 표준과 사실상 같은 조항)까지 전부 "이탈"로 표시됩니다.

## 2. 왜 그런가 — 원인 두 가지

### 원인 A — 이탈 판정이 "글자가 얼마나 똑같은지"로 NONE/CHANGED를 가르는데, 이 신호로는 구분이 안 됨

**지금 로직:** 사용자 조항과 표준 조항의 **글자 단위 일치율**(파이썬 `difflib.SequenceMatcher`, 공백 제거 후 문자 비교)을 재서, 0.85 이상이면 "변경 없음(NONE)", 미만이면 "변경됨(CHANGED)"으로 판정합니다. ([core/deviation.py:45](../../core/deviation.py#L45))

**무엇이 문제인가 (골든 92건으로 실제 계산):**

| 정답 라벨 | 글자 일치율 중앙값 | 최댓값 | 0.85 넘은 개수 |
| --- | --- | --- | --- |
| NONE (변경 없음) 43건 | 0.196 | 0.703 | **0** |
| CHANGED (변경됨) 43건 | 0.102 | 0.323 | 0 |

두 가지가 동시에 보입니다.
1. **NONE인데도 일치율이 0.85를 절대 못 넘습니다**(최대 0.703). 그래서 현재 기준(0.85)으로는 "변경 없음" 판정이 **원천적으로 불가능** → 매칭된 조항은 죄다 CHANGED.
2. 더 근본적으로, **NONE 그룹과 CHANGED 그룹의 일치율 분포가 심하게 겹칩니다**(둘 다 대부분 0.02~0.3대). 기준값을 어디로 옮겨도 둘을 깨끗이 못 가릅니다.

**왜 이렇게 낮게 나오나?** 골든의 `user_clause`는 **한 문장(단문)**인데, 비교 대상 표준은 **"조(條)" 전체(여러 항)**입니다. 길이·형식이 크게 달라서, 내용이 같든 다르든 글자 일치율이 바닥에 깔립니다. 즉 지금 지표는 "내용이 바뀌었나"가 아니라 사실상 "길이가 다른가"를 재고 있습니다.

**그럼 임베딩(의미 유사도)로 바꾸면 되지 않나? → 아니오. (아래 3장에서 자세히)**

### 원인 B — 독소 탐지가 조항마다 무조건 5개를 붙임 (100% 오탐) → **수정 완료**

**지금 로직:** 각 조항을 독소패턴 DB에서 검색해 **가장 가까운 5개(top-5)**를 가져오고, 리랭커 점수(sigmoid, 0~1)가 임계값 이상이면 "독소"로 답니다. ([review_pipe.py](../../pipe/review_pipe.py))

**진짜 원인 (점수 분포를 직접 떠서 확인 — 처음 추측과 달랐음):**
독소 후보의 리랭커 점수를 gold_toxic 유무별로 떠 보니, **정상(clean) 조항의 최고 후보 점수가 대부분 정확히 `0.500`** 이었습니다. `0.500`은 `sigmoid(0)`, 즉 **리랭커가 "전혀 관련 없음(로짓 0)"이라고 답한 바닥값**입니다. 그런데 임계값이 `0.5`이고 필터가 `점수 >= 0.5`라, **무신호(0.500)까지 전부 통과** → 조항마다 5개 다 달림. "점수가 높아서"가 아니라 **임계값이 sigmoid 바닥과 겹쳐서** 생긴 축퇴였습니다. (게다가 같은 패턴이 여러 청크로 중복돼 top-k를 같은 값으로 채우는 문제도 있었음)

임계값 스윕(진짜 독소 vs 정상 분리 성능):

| 임계값 | recall(진짜 독소 잡음) | 특이도(정상 통과) |
| --- | --- | --- |
| 0.50 (기존) | 1.000 | **0.000** ← 다 찍음 |
| 0.60 | 0.385 | 0.864 |
| 0.70 | 0.346 | 0.909 |
| 0.80+ | 0.000 | 1.000 ← 다 놓침 |

**적용한 수정 (3가지):**
1. **중복 제거**([core/toxic.py](../../core/toxic.py)): 같은 패턴은 최고점 1회만.
2. **임계값 0.5 → 0.6**([review_pipe.py](../../pipe/review_pipe.py) `toxic_threshold`): sigmoid 바닥 위로 올려 무신호 통과 차단.
3. **독소 전용 후보 수 분리**(`toxic_top_k=3`): 나머지는 바닥 노이즈라 소수만.

**수정 효과 (v1 골든 기준, 예상):** 독소 특이도 **0.000 → 0.864**, precision 0.28~0.35 → **0.53**. 단 **recall은 1.0 → 0.385로 하락**합니다 — 이는 임계값 탓이 아니라 **리랭커가 진짜 독소 상당수에도 바닥(0.5) 점수만 주는 근본 한계** 때문입니다(임계값만으론 recall·특이도 동시 확보 불가).

> ⚠️ **남은 과제(v2):** 독소 탐지의 recall은 리랭커/독소 코퍼스의 매칭력에 막혀 있습니다. 임계값 튜닝이 아니라 **독소 패턴 코퍼스 보강·매칭 방식 개선**이 필요합니다. 또한 임계값 0.6은 v1 92건 분포로 잡은 **시작값**이므로 확대 골든셋으로 재보정해야 합니다.

## 3. "임베딩으로 바꾸자"를 철회하는 이유 (법률 도메인·과적합 관점)

처음엔 "글자 비교 말고 의미(임베딩) 유사도로 NONE을 판정하자"고 제안했지만, **철회**합니다. 팀에서 오해 없도록 근거를 남깁니다.

1. **법률 문서에서 위험한 변경은 '의미상 거의 똑같은데 법적으로 정반대'인 경우가 많습니다.**
   - "지연 시 이자를 부과**한다**" ↔ "부과하지 **아니한다**" (부정어 하나)
   - "손해액의 **10%**" ↔ "**50%**" (숫자)
   - "**갑**이 부담" ↔ "**을**이 부담" (당사자)
   임베딩은 이런 쌍을 "거의 같다(=NONE)"로 뭉갭니다 → **가장 중요한 이탈을 놓칩니다(FN).** 그래서 [core/README.md](../../core/README.md)가 "글자 단위 비교"를 고른 **방향 자체는 옳습니다.** 문제는 방향이 아니라 **비교 단위**(단문 vs 조 전체)입니다.

2. **지금의 과탐은 '안전한 실패'입니다.** 우리 서비스는 "위법이다"가 아니라 **"검토해 보세요" 후보를 띄우는 것**(AGENTS #3)이라, 이탈을 **놓치는 것(FN)**이 잘못 알리는 것보다 훨씬 위험합니다. v1은 이탈 FN=0(하나도 안 놓침)이라 **안전 측면은 오히려 양호**합니다. 특이도=0은 "다 빨간불이라 우선순위를 못 준다"는 **유용성** 문제지, 안전 문제는 아닙니다. (단, 독소 100% 오탐은 유용성을 아예 무너뜨리므로 예외적으로 시급)

3. **정답(골든)이 아직 확정 안 됐습니다.** v1 골든은 **말바꿈(paraphrase)을 NONE**으로 라벨했는데, README 철학은 "의미가 같아도 문구가 바뀌면 CHANGED"라 **정반대**입니다. 즉 **특이도=0이 '시스템 버그'인지 '골든 라벨이 시스템 철학과 안 맞는 것'인지 아직 못 가립니다.** → **"변경 없음(NONE)"의 정의를 사람이 먼저 확정**해야 그 다음이 정해집니다.

4. **작은 골든셋에 맞춰 기준값을 튜닝하면 과적합입니다.** 92건(특히 SW 17건)에 특이도가 좋아지도록 threshold를 맞추면, 그 값은 **이 예시 데이터에만** 맞고 실제 계약서엔 안 맞을 가능성이 큽니다.

## 4. v1 골든셋 — 강점 (유지할 것)

- **정상(참음성) 케이스를 넣었다.** `gold_deviation=NONE`·`trap=none`이 있어서 **특이도를 잴 수 있었고**, 그 덕분에 "다 이탈로 찍는 축퇴"가 드러났습니다. 이 라벨이 없었으면 Recall=1.0에 가려 못 봤을 문제입니다. → **가장 큰 공헌.**
- **함정 유형을 명시**(`paraphrase` 말바꿈 / `contradiction` 내용 반대 / `partial` 부분변경 / `none`)해, 검색이 어떤 함정에 강한지 유형별로 해석할 수 있습니다.
- **세 가지 라벨 완비**: 검색 정답(`gold_clause_id`) + 이탈 3분류(`gold_deviation`) + 독소(`gold_toxic`) → 한 데이터로 세 지표를 동시에 채점.
- **3개 계약유형**(SI/SM/SW)을 실제 표준 코퍼스와 연결해 커버.

## 5. v1 골든셋 — 약점 (v2에서 고칠 것)

- **표본이 작고 불균형.** SW는 17건뿐이고, 특이도의 분모(정상 케이스)가 한 자릿수라 지표가 몇 건에 크게 흔들립니다. 통계적으로 믿기 어렵고 튜닝 시 과적합 위험.
- **"변경 없음(NONE)"의 정의가 모호.** 말바꿈을 NONE으로 본 라벨이 시스템 철학과 충돌합니다. 정의를 글로 못 박지 않아 "정답"이 흔들립니다. (→ 원인 A가 버그인지 라벨 문제인지 못 가리는 핵심 이유)
- **정상=독소없음이 암묵적.** `gold_toxic`이 선택 필드라, 무해한 조항에 "독소 없음"이 명시돼 있지 않습니다. 특이도 분모가 추정에 의존.
- **비교 단위 불일치.** `user_clause`(단문)와 표준(조 전체)의 길이가 달라 이탈 판정 신호를 왜곡합니다(원인 A). 골든이 "조 단위 대응"을 의도한 건지 "항 단위"인지 설계 의도가 불명확.
- **id 규약이 없었음.** `gold_clause_id`가 DB의 실제 id(버전 세그먼트·자릿수 패딩)와 안 맞아 v1에서 임시로 맞췄습니다. SI는 표준이 2022·2025 두 버전이라 **임시로 2022**를 썼습니다 — 정식 규약 문서화 필요.
- **독소 라벨이 단순.** 조항당 독소를 유무/단일로만 봐서 복수 패턴·심각도(severity)를 반영하지 못합니다.
- **누락(MISSING) 평가 없음.** "표준엔 있는데 계약서에 통째로 빠진 조항"은 계약 단위(트랙 B)라야 재는데, v1엔 없습니다.

## 6. v2 만들 때 체크리스트

- [ ] **"변경 없음(NONE)"의 정의부터 확정**: 엄격안(글자·서식만 같으면 NONE, 말바꿈은 CHANGED) vs 관대안(의미가 같으면 NONE). 정한 뒤 말바꿈 케이스를 그에 맞게 재라벨.
- [ ] **모든 케이스에 `gold_toxic` 명시**(독소 없으면 빈 배열로라도) — 정상 케이스를 확실히 표시.
- [ ] **표본 확대 + 유형별 균형**, 그리고 **튜닝셋/검증셋 분리**(held-out)로 과적합 방지.
- [ ] **`gold_clause_id` 규약 문서화**: 버전 세그먼트 포함 여부, 자릿수 패딩, SI 버전(2022/2025) 선택 기준.
- [ ] **법적 치명 변경을 별도 함정으로 분리**: `negation`(부정어)·`number`(금액·기간)·`party`(당사자). 임베딩이 놓치는 케이스를 눈에 보이게 측정.
- [ ] (검토) **항(項) 단위 라벨** 추가 — 비교 단위 불일치(원인 A) 해소용.
